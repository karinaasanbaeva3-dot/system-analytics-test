# Задание 3: Архитектура PUSH-уведомлений

## Схема 1: Регистрация устройства и получение push-токена

1. Пользователь открывает мобильное приложение
2. Приложение запрашивает разрешение на показ уведомлений
3. Пользователь нажимает "Разрешить"
4. Приложение получает push-токен от FCM (Android) или APNS (iOS)
5. Приложение отправляет push-токен на бэкенд
6. Notification Service сохраняет токен в Token Registry (связка user_id → push_token)

## Схема 2: Отправка push-уведомления при событии (например, отмена заказа)

### Шаг 1. Возникновение события
- Cart Service: "Корзина брошена" (пользователь не оформил заказ за 1 час)
- Order Service: "Заказ отменен"
- Promo Service: "Новая рекламная акция"

### Шаг 2. Передача события в очередь
Сервис-источник публикует событие в Event Bus (Kafka / RabbitMQ)

### Шаг 3. Обработка события Notification Service
1. Notification Service получает событие из Kafka
2. Определяет тип события (отмена заказа / реклама / корзина брошена)
3. Формирует текст уведомления из шаблона (например: "Ваш заказ №123 отменен")
4. Ищет push_token пользователя в Token Registry
5. Определяет платформу пользователя (Android → отправка в FCM, iOS → отправка в APNS)
6. Отправляет запрос во внешний push-сервис (FCM или APNS)

### Шаг 4. Доставка на устройство
- FCM или APNS доставляет уведомление на устройство пользователя
- Приложение получает уведомление и отображает его в шторке
- Пользователь нажимает на уведомление → открывается соответствующий экран

## Схема 3: Обработка ошибок при отправке push

- Попытка 1: отправка в FCM/APNS
  - Если успешно → уведомление доставлено (конец)
  - Если ошибка → переход к попытке 2

- Попытка 2: повтор через 5 секунд
  - Если успешно → уведомление доставлено (конец)
  - Если ошибка → переход к попытке 3

- Попытка 3: повтор через 5 секунд
  - Если успешно → уведомление доставлено (конец)
  - Если ошибка → уведомление попадает в Dead Letter Queue

- Dead Letter Queue:
  - Уведомление сохраняется для ручного разбора
  - Отправляется алерт в систему мониторинга (Sentinel / Grafana)
  - (опционально) Fallback: отправка SMS или email

## Типы PUSH-уведомлений

| Тип уведомления | Приоритет | Время отправки |
|----------------|-----------|----------------|
| Отмена заказа | Высокий | Сразу |
| Статус доставки | Высокий | Сразу |
| Подтверждение оплаты | Высокий | Сразу |
| Корзина брошена (abandoned cart) | Средний | Через 1 час бездействия |
| Рекламная рассылка | Низкий | По расписанию (например, в 9:00 и 20:00) |

## Компоненты архитектуры

| Компонент | Назначение |
|-----------|------------|
| Notification Service | Главный сервис отправки пушей. Формирует уведомления из шаблонов, управляет очередью. |
| Token Registry | Хранит push-токены пользователей. Связка: user_id → push_token, платформа, часовой пояс. |
| Schedule Service | Планировщик для рекламных и периодических пушей. Работает на основе cron-выражений. |
| Event Bus (Kafka) | Очередь событий. Другие микросервисы публикуют события сюда. |
| FCM (Firebase) | Внешний сервис Google для доставки push на Android. |
| APNS (Apple Push) | Внешний сервис Apple для доставки push на iOS. |
